# House style

Quick reference. Defer to the file you're editing — match its conventions
before applying any of these.

## mozilla-central (C++ / JS / Rust / Python)

- Run `./mach lint` before sending. Fix what it complains about; don't
  argue with it in chat.
- Commit message: `Bug NNNNNN - <short summary>. r=<reviewer>`
- Don't include a bug number you haven't filed yet. File the bug first.
- No `// Generated by …` headers. No `// TODO(micah):` comments. If a TODO
  is real, it goes in Bugzilla.
- Comments explain *why*, never *what*. If the code is opaque, fix the code.
- Push to try before submitting on anything non-trivial; link the try run
  in the Phabricator description.

## Firefox-AI / dataservices-infra (TypeScript / Python / Helm)

- Match the file. If the file has semicolons, you have semicolons.
- Helm values changes go through `atlantis apply` then merge. Don't merge
  before the plan succeeds.
- For k8s ops (`kubectl scale`, etc.), prefer editing the values file and
  letting Argo reconcile over imperative commands. Imperative commands are
  for incidents, not changes.
- New env vars: add to `values-{prod,stage,nonprod}.yaml`, document in the
  service README.

## Phabricator review comments

- Quote the line you're pointing at when it's not obvious.
- Suggest a concrete change, not just "this is wrong."
- One concern per comment. If you have five concerns, leave five comments,
  not one nested list.
- Link prior art (other revs, blog posts, RFCs) when it would actually help.
  Don't link Wikipedia.

## Slack

- Lowercase by default. Sentence case is fine if it's a status update.
- Threads, not channel walls. Keep top-of-channel for status and decisions.
- If you @-mention someone, finish the thought in the same message. Don't
  ping then pause.
- Emoji reactions, not emoji prose, unless someone else is using them first.

## Bugzilla

- One thought per comment. Multi-paragraph dumps are hard to follow on
  Bugzilla's threading.
- Always link: try run, Phabricator rev, Jira issue, and any related bugs.
- Whiteboard labels (`[aife]` / `[aimodels]` / `[aiplatform]`) are set by
  the TL/EM. Don't set Priority in Bugzilla — that lives in Jira.

## What never goes in any artifact

- Credentials, tokens, cookies, session keys, IPs of internal hosts.
- Sentences that exist only to be polite ("I hope this helps!", "Great work
  overall but…", "Please let me know if you have any questions"). Mozilla
  reviewers find them noisy.
- Marketing voice. Plain technical prose, always.
